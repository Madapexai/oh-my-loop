# Oh My Loop 工作原理

## 问题

AI agent 的失败方式是可预测的。最常见的是**过度循环**：把每个任务都包在多步推理循环里，因为"循环很强大"。第二常见的是**验证不足**：声称任务完成却不跑任何检查。

Oh My Loop 解决这两个问题：给 agent 一个路由器决定**是否需要循环**，并强制模式在声称完成前**必须验证**。

## 路由器

[using-oh-my-loop](../../using-oh-my-loop/SKILL.md) 是入口。它跑一个决策树：

1. 任务是否平凡？ -> 直接回答，停止
2. 是否可逆且低风险？ -> 做一次，停止
3. 是否有可验证的成功标准？ -> 没有，先定义
4. 是否复杂到第一次可能失败？ -> 不复杂，执行+验证，停止
5. 按失败模式选 pattern

这阻止了过度循环。大多数任务到第 2 步就停了。

## Patterns

五个 pattern，按失败模式选（不是按偏好）：

- **react** - 不知道步骤，需要探索
- **reflexion** - 第一次大概率错，可验证
- **plan-execute** - 步骤已知，执行可能错
- **self-refine** - 输出需要打磨，不是修复
- **multi-agent** - 需要多视角

每个 pattern 有：
- **终止条件** - 没有无限循环
- **检查点** - 进入、退出、失败、升级
- **约束** - 成本、时间、降级路径
- **完整示例**

## 组件

四个可组合的部件，用于自定义循环：

- **verify-before-claim** - 阻止未验证的声称
- **task-decomposition** - 把任务拆成可验证的子任务
- **feedback-loop** - 记录结果，改进未来运行
- **self-questioning** - 承诺前的多视角检查

## 三个优先级

每个任务在**效果、成本、效率**之间权衡。路由器按上下文决定：

| 任务类型 | 优先级 | 默认行为 |
|---|---|---|
| 生产 bug 修复、不可逆操作 | 效果优先 | reflexion + verifier |
| 一次性草稿、原型 | 效率优先 | 单次 + self-refine |
| 昂贵 API、批处理 | 成本优先 | 便宜模型、最少轮次 |
| 探索、调研 | 效果优先 | react、多轮 |

路由器不会让用户选。它按任务决定。

## 人工介入

只有三种情况暂停确认：

1. **不可逆操作** - 删除、部署、付款、群发邮件
2. **用户数据修改** - 改数据库、写用户存储
3. **成本超阈值** - 超预算

其他全部 agent 自己决定并执行。

## 设计你自己的循环

如果五个 pattern 都不合适，用 [write-a-loop](../../write-a-loop/SKILL.md)。它强制 5 步硬流程：

1. **定义目标** - 包含至少 3 个失败模式
2. **选 pattern** - 按主要失败模式
3. **定义检查点** - 进入、退出、失败、升级
4. **定义约束** - 预算、不可逆、降级路径
5. **写并测试** - 测试你自己的失败模式

硬规则：**列不出失败模式，说明你还不理解任务。**

## 这不是什么

- 不是 agent 框架（没有 runtime、没有编排引擎）
- 不是 prompt 库（没有现成 prompt）
- 不是公司 SOP（没有内部工具或专有名）
- 不是"一切皆循环"的包装

它是一套方法论：patterns + components + examples，你组合成适合你任务的循环。
